## 3. 任务状态口径（已查证）

为避免“当前任务列表”定义歧义，会话中已基于 Blizzard API 文档与现有代码确认以下口径可用：

1. `C_QuestLog.IsQuestFlaggedCompleted(questID)`：是否已完成
2. `C_QuestLog.GetLogIndexForQuestID(questID)`：是否在任务日志（进行中判断）
3. `C_QuestLog.ReadyForTurnIn(questID)`：是否可交付

建议“当前任务列表”优先口径：

1. 仅展示 `ReadyForTurnIn + Active`
2. 列表排序为“可交付优先，再进行中”
3. 已完成不进入“当前任务”列表

注：最终是否采用该口径，仍待用户最终拍板。

## 4. 方案池（本轮已讨论）

### 方案1：双栏渐进式（稳健型）

1. 左树（分类树）+ 右侧列表/详情二态切换
2. 改动最小、学习成本低
3. 后续扩展时右侧可能拥挤

### 方案2：三栏工作台（推荐）

1. 左栏分类树 + 中栏当前任务 + 右栏详情与操作
2. 最适配后续导航/高亮扩展
3. 职责边界清晰，产品形态稳定

### 方案3：地图导向主屏（探索型）

1. 中央强调地图上下文与热区
2. 任务树降级为二级入口
3. 改造幅度较大，用户习惯迁移成本高

### 方案4：页签工作台（同屏分层）

1. 左树保留，右区拆为页签：当前任务/详情/导航
2. 实施节奏稳健，复杂度适中
3. 高频切换时操作步数略增

### 方案5：状态看板（执行导向）

1. 以状态泳道组织：可交付/进行中/待解锁
2. 强调“当前该做什么”
3. 与传统任务树心智有差异，需要适应

### 方案6：焦点模式（单任务深潜）

1. 单任务主视图 + 轨道切换
2. 强执行、弱检索
3. 不适合密集浏览场景

## 6. 建议的收敛方向（面向 2/4/5 可切换）

建议采用统一框架：

1. **同一数据模型**：任务树、当前任务、详情、导航动作共享同一状态源
2. **多视图切换**：`Workbench(2)` / `Tabs(4)` / `Board(5)` 仅切换布局，不切换业务能力
3. **统一能力入口**：地图导航、NPC 导航、位置高亮统一挂载于任务操作区
4. **统一持久化**：记录默认视图模式与各视图局部状态（滚动位、筛选、折叠等）

该方向可确保后续扩展时不因 UI 视图分叉导致能力重复开发。
