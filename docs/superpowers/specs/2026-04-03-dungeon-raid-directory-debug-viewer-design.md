# 地下城 / 团队副本目录调试查看器 — 设计规格

**状态**：已实现（首版已落地，文档按当前代码回填）。  
**日期**：2026-04-03  
**范围**：在设置页现有“地下城 / 团队副本目录缓存”区块下，新增一个可滚动的只读文本调试视图，用于查看 `Toolbox.DungeonRaidDirectory` 当前缓存与运行时覆盖层的完整快照。  
**定位**：开发 / 排查优先，不追求最终用户向的信息架构或视觉表现。

---

## 1. 目标与验收

| 项 | 说明 |
|----|------|
| **入口** | 位于 `/toolbox` 设置页中“地下城 / 团队副本目录缓存”区块之下。 |
| **形态** | 一个固定高度、可滚动、只读的多行文本区域。 |
| **数据范围** | 展示目录缓存的**全局摘要** + **仅含坐骑掉落的副本记录** + **运行时锁定覆盖层摘要**。 |
| **展示风格** | 纯文本调试快照，稳定顺序，便于快速复制/对照/截图排查。 |
| **刷新能力** | 至少提供“刷新快照”按钮；设置页打开时允许低频自动刷新。 |
| **未知语义** | `nil` / 未完成扫描必须原样体现，不得伪装为 `false`。 |
| **边界** | 首版不做搜索、不做过滤、不做展开折叠、不做复制按钮。 |

**首版验收建议**

1. 打开 `/toolbox` 后，能在目录缓存区下方看到可滚动文本视图。  
2. 当目录正在构建时，快照中的状态、阶段、进度与部分副本记录会随刷新而更新。  
3. 能看见每个含坐骑掉落副本的基础信息、难度列表、坐骑摘要与锁定摘要。  
4. 未完成扫描的难度 / 摘要明确显示为 `nil` 或等价文本，而不是被静默省略。  
5. 即使文本很长，也不会把整个设置页撑爆；查看依赖内部滚动区域完成。

## 3. 实现方向对比

| 方案 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| **A. 只读文本快照浏览器**（**已选**） | 目录层生成多行字符串，设置页放滚动文本区显示 | 实现简单、排查效率高、边界清晰 | 文本长时需要重建整块字符串 |
| B. 伪结构化文本列表 | 每个副本渲染一组 FontString/块级容器 | 阅读体验稍好 | UI 对象更多，性能与复杂度更高 |
| C. 导出型面板 | 只做按钮，把内容输出到聊天框或弹窗 | 最省实现 | 不满足“设置中滚动查看”的目标 |

**采用 A**：让目录层统一负责“调试快照的内容与格式”，Settings 只负责承载与滚动显示。

---

## 4. 架构边界

### 4.1 目录层负责内容

调试查看器的数据与文本格式由 `Toolbox.DungeonRaidDirectory` 自己提供，避免 `SettingsHost` 了解过多目录内部结构。

建议新增两个对外接口：

- `Toolbox.DungeonRaidDirectory.GetDebugSnapshot()`
  - 返回结构化调试快照表，便于未来复用
- `Toolbox.DungeonRaidDirectory.FormatDebugSnapshot()`
  - 返回最终显示用的多行字符串，供设置页直接显示

### 4.2 SettingsHost 负责承载

`Toolbox.SettingsHost` 只负责：

- 创建滚动区域
- 放置“刷新快照”按钮
- 在面板可见时低频刷新文本
- 控制固定高度与滚动行为

`SettingsHost` 不应自己遍历 `records` 拼字段，以免未来目录层字段调整时 UI 跟着散改。

---

## 5. 快照内容建议

### 5.1 全局摘要

快照顶部建议先输出：

- `state`
- `currentStage`
- `completedUnits / totalUnits`
- `lastBuildAt`
- `recordCount`
- `supportedDifficultyCount`
- `mountPositiveRecordCount`
- `lockoutMappedCount`

### 5.2 单副本段落

每个副本建议按稳定顺序输出：

```text
[1190] 奈幽巴宫殿
  kind=raid tierIndex=10 mapID=2215 worldInstanceID=2648
  summary.hasAnyMountLoot=true mountDifficultyIDs=15,16
  difficulties:
    14 普通 hasMountLoot=false lockout=nil
    15 英雄 hasMountLoot=true lockout={ reset=123456 progress=6/8 extended=false }
```

### 5.3 文本规则

- 按 `runtime.recordOrder` 输出，保证顺序稳定  
- 同一副本内部按 `difficultyOrder` 输出  
- `nil` 保留为显式文本  
- 空列表显示为 `[]` 或 `none`，避免看起来像漏字段

## 6. UI 方案

### 6.1 位置

放在当前“地下城 / 团队副本目录缓存”区块内部，位于：

1. 状态 / 阶段 / 进度条  
2. 调试聊天输出开关  
3. **调试快照查看区**

### 6.2 组成

建议包含：

- 小标题：如“目录快照”
- “刷新快照”按钮
- 固定高度滚动容器
- 只读多行 `EditBox` 或等价只读文本承载控件

### 6.3 刷新策略

首版建议：

- 打开设置页后，手动点击按钮可立即刷新
- 面板显示期间每 `0.2 ~ 0.5` 秒自动刷新一次
- 若文本未变化，可跳过 `SetText`，减少抖动与无效更新

---

## 7. 边界与非目标

- 首版**不做**搜索框  
- 首版**不做**副本过滤（按资料片 / 类型 / 是否有坐骑）  
- 首版**不做**复制到剪贴板按钮  
- 首版**不做**卡片化 / 表格化展示  
- 首版**不要求**和聊天调试输出完全同格式；两者可以共享底层快照，但展示层允许不同

---

## 8. 风险

| 风险 | 说明 | 缓解 |
|------|------|------|
| 文本过长 | 全量记录可能很长 | 固定高度 + 独立滚动区域 |
| 频繁刷新 | 构建中多次 `SetText` 可能带来卡顿 | 低频刷新 + 文本未变化时跳过 |
| UI 层耦合目录结构 | SettingsHost 若自己拼数据，后续字段变更会很脆 | 由目录层统一提供快照接口 |
| `nil` 误处理 | 直接 tostring 容易和空串混淆 | 明确保留 `nil`、`[]`、`none` 等文本 |

## 10. 设计结论

本需求首版应作为 `Toolbox.DungeonRaidDirectory` 的**调试查看能力**落地：  
在设置页现有目录缓存区块中增加一个固定高度、可滚动、只读的文本快照面板，数据由目录层统一生成并格式化。  
该方案以开发排查效率优先，最小化 UI 复杂度，同时为后续的搜索、过滤、导出能力保留清晰扩展点。
