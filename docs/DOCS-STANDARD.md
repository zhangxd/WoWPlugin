# docs 文档规范

> 本文件是 `docs/**` 下功能、需求、设计、计划、测试文档的唯一写作规范。凡新增或修改这些类型的文档，必须遵循本文件；若与旧写法冲突，以本规范为准。

---

## 1. 适用范围

本规范适用于以下文档：

- `docs/features/*.md` 功能说明
- `docs/specs/*.md` 需求规格
- `docs/designs/*.md` 设计方案
- `docs/plans/*.md` 实施计划
- `docs/tests/*.md` 测试计划与测试记录
- 以上文档在后续迭代中的新增、改写、迁移与归档

本规范不替代以下固定职责文档，但这些文档在引用功能、需求、设计、计划、测试时，必须指向符合本规范的文件：

- `docs/Toolbox-addon-design.md`：总体架构与长期设计单一事实来源
- `docs/AI-ONBOARDING.md`：AI 协作读档顺序与执行路径
- `docs/FEATURES.md`：产品能力总览
- `docs/release.md`：发布说明与发版流程

---

## 2. 强制规则

1. 一个文档只能有一个主类型：功能、需求、设计、计划、测试五选一；禁止把多种主类型混写在同一文件中。
2. 新增文档前，必须先确定文档类型、目录落点、文件名和模板；禁止先写内容、再决定放哪里。
3. 新增文档必须使用 `docs/templates/` 下对应模板起稿；允许删减不适用小节，但不得省略元信息区块。
4. 每份文档都必须写明状态、范围、关联模块或目录、关联文档、最后更新日期。
5. 同一 `topic` + `type` 只能维护一份文档；同类型文档不再通过日期或版本尾巴区分，变更追溯依赖 Git 历史。
6. 同一功能在同一文档类型下只保留当前有效的一份文档；后续重构、补充、澄清、状态推进都直接修改原文件，不得为同一功能另起“重构版”“续篇”“第二版”等平行文件。
7. 功能、需求、设计、计划、测试文档必须能串起来：功能说明描述“现在能用什么”，设计指向需求或现状实现，计划指向需求或设计，测试指向功能/需求/设计/计划中的至少一项。
8. 影响长期架构、模块映射、DB 结构、TOC 或公共 API 的设计落地后，必须回写 `docs/Toolbox-addon-design.md`；局部设计文档不能代替总设计。
9. `docs/**` 中禁止新增语义不明的临时文件名，如 `notes.md`、`temp.md`、`new.md`、`杂项记录.md`。
10. 历史遗留目录可以保留存量文件，但不得继续新增同类文档；若对存量文件做实质性续写，优先迁移到规范目录。

## 3. 目录落位规则

### 3.1 现行有效目录

| 路径 | 角色 | 允许内容 |
|------|------|----------|
| `docs/features/` | 功能说明 | 当前已实现能力、入口、设置、限制、关联模块与相关文档 |
| `docs/specs/` | 需求规格 | 背景、范围、确认决策、待确认项、验收标准、状态 |
| `docs/designs/` | 设计方案 | 目标、非目标、方案对比、选定方案、影响面、风险、验证策略 |
| `docs/plans/` | 实施计划 | 输入文档、影响文件、执行步骤、验证命令、回滚与交付记录 |
| `docs/tests/` | 测试文档 | 测试范围、环境、用例、执行结果、缺陷与阻塞 |
| `docs/templates/` | 模板库 | 标准模板，供新文档复制使用 |

### 3.2 固定职责文件

| 路径 | 角色 | 禁止承载的内容 |
|------|------|----------------|
| `docs/Toolbox-addon-design.md` | 总体设计总纲 | 单次需求讨论、逐步实施计划、测试执行记录 |
| `docs/AI-ONBOARDING.md` | 协作与读档规则 | 单次功能方案细节、测试明细 |
| `docs/FEATURES.md` | 产品功能总览 | 单次需求、单次设计推导、实施计划、测试记录、开发接入说明、示例代码；它只做全局索引，不代替 `docs/features/*.md` 的模块级功能说明 |
| `docs/release.md` | 发布记录与发布流程 | 需求澄清、实现细节设计 |

### 3.3 历史遗留目录

以下目录保留存量文件，但从本规范生效后禁止新增同类文档：

| 路径 | 现状 | 处理规则 |
|------|------|----------|
| `docs/superpowers/specs/` | 曾存放局部设计/规格 | 不再新增；新的需求或设计统一进入 `docs/specs/` 或 `docs/designs/` |
| `docs/superpowers/plans/` | 曾存放局部实施计划 | 不再新增；新的计划统一进入 `docs/plans/` |

若历史文件被继续使用，应在下一次实质性修改时迁移为新命名和新目录。

## 4. 命名规则

### 4.1 文件命名格式

功能、需求、设计、计划、测试文档统一使用：

```text
<topic>-<type>.md
```

其中：

- `<topic>`：稳定主题名，使用小写字母、数字、连字符；优先包含模块 id 或主题关键词
- `<type>`：固定后缀，只能是 `features`、`spec`、`design`、`plan`、`test`

示例：

- `encounter-journal-features.md`
- `encounter-journal-quest-tab-performance-spec.md`
- `encounter-journal-quest-tab-performance-design.md`
- `encounter-journal-quest-tab-performance-plan.md`
- `encounter-journal-quest-tab-performance-test.md`

### 4.2 命名补充规则

1. 同一主题在不同阶段应共享同一个 `<topic>`，靠后缀区分文档类型。
2. 同一 `topic` + `type` 只保留一份文档，不再新增日期前缀、`v2`、`final-final` 等变体文件；历史追溯依赖 Git 历史。
3. 若同一功能进入重构、补录规则、补验收、补测试等后续阶段，且主题边界未变化，必须继续修改原有 `<topic>-<type>.md`；禁止仅因“这次是重构”就新建平行文档。
4. 若主题实质变化，应新建新 `topic` 文档，而不是在原文件名后继续追加版本尾巴。
5. 禁止使用中文空格、括号、无意义版本尾巴作为主题名，例如 `new`、`latest`、`final-final`。

## 5. 元信息与状态

### 5.1 顶部元信息区块

每份规范文档标题下方必须紧跟如下元信息区块：

```markdown
- 文档类型：功能 / 需求 / 设计 / 计划 / 测试
- 状态：<按文档类型选择一个合法状态>
- 主题：<topic>
- 适用范围：<模块、目录、功能范围>
- 关联模块：<moduleId / 无>
- 关联文档：
  - <相对路径或 无>
- 最后更新：YYYY-MM-DD
```

规则：

1. `文档类型` 只能填一个。
2. `主题` 必须与文件名中的 `<topic>` 一致。
3. `关联文档` 至少写出直接上游文档；无则明确写 `无`。
4. `最后更新` 每次改文档都要改。

### 5.2 状态枚举

| 文档类型 | 合法状态 |
|----------|----------|
| 功能 | `草稿` / `已发布` / `已归档` |
| 需求 | `草稿` / `待确认` / `可执行` / `已完成` / `已归档` |
| 设计 | `草稿` / `待评审` / `已确认` / `已落地` / `已归档` |
| 计划 | `草稿` / `待执行` / `执行中` / `已完成` / `已归档` |
| 测试 | `草稿` / `待执行` / `执行中` / `已通过` / `有问题` / `已归档` |

状态使用规则：

1. 已实现并对外可见的功能说明文档，状态应写为 `已发布`；不再对外维护时改为 `已归档`。
2. 用户明确回复“开动”且确认结果已写回需求文档后，需求文档状态改为 `可执行`。
3. 设计评审通过后，设计文档状态改为 `已确认`；设计真正落地并回写总设计后，改为 `已落地`。
4. 计划开始执行时改为 `执行中`；完成验证后改为 `已完成`。
5. 测试文档执行完毕后，按结果写成 `已通过` 或 `有问题`；禁止执行完后仍保持 `草稿`。

## 6. 各类型必备章节

### 6.1 功能文档

功能文档至少包含以下章节：

1. `定位`
2. `适用场景`
3. `当前能力`
4. `入口与使用方式`
5. `设置项`
6. `已知限制`
7. `关联文档`
8. `修订记录`

要求：

- 功能文档描述“现在已经能用什么”，不能写成待实现方案。
- 若某项能力是跨模块联动实现，必须写明主归属模块与协作模块。
- 功能文档可以链接到 `docs/FEATURES.md`，但不能被 `FEATURES.md` 取代。

### 6.2 需求文档

需求文档至少包含以下章节：

1. `背景`
2. `目标`
3. `范围`
4. `已确认决策`
5. `待确认项`
6. `验收标准`
7. `实施状态`
8. `修订记录`

要求：

- `范围` 必须同时写清“做什么”和“不做什么”。
- `已确认决策` 与 `待确认项` 必须分开写，禁止混在一处。
- `验收标准` 必须是可验证结果，不能写成实现动作。

### 6.3 设计文档

设计文档至少包含以下章节：

1. `背景`
2. `设计目标`
3. `非目标`
4. `方案对比`
5. `选定方案`
6. `影响面`
7. `风险与回退`
8. `验证策略`
9. `修订记录`

要求：

- `方案对比` 至少写 2 个方案；若只有 1 个方案，应说明为什么没有可行替代。
- `影响面` 至少覆盖数据、API、目录/文件、文档回写中的相关项。

### 6.4 计划文档

计划文档至少包含以下章节：

1. `目标`
2. `输入文档`
3. `影响文件`
4. `执行步骤`
5. `验证`
6. `风险与回滚`
7. `执行记录`
8. `修订记录`

要求：

- `执行步骤` 使用复选框，粒度以“一个可验证动作”为单位。
- `验证` 必须写具体命令、检查动作或游戏内验证点。
- 计划文档不能替代需求或设计；上游文档必须在 `输入文档` 中列出。

### 6.5 测试文档

测试文档至少包含以下章节：

1. `测试背景`
2. `测试范围`
3. `测试环境`
4. `测试用例`
5. `执行结果`
6. `问题与阻塞`
7. `结论`
8. `修订记录`

要求：

- `测试用例` 至少包含编号、前置条件、操作、预期结果。
- `执行结果` 必须记录实际结果，而不是只重复预期结果。
- 若存在阻塞，必须明确说明阻塞原因与未完成范围。

## 7. 关联与回写规则

1. 模块级功能说明应写入 `docs/features/<topic>-features.md`；`docs/FEATURES.md` 只做全局产品总览与入口索引。
2. 新需求文档建立后，如进入方案阶段，应新增对应设计文档，而不是在需求文档里直接混写实现细节。
3. 设计确定后，如要执行，应新增对应计划文档；计划只负责“怎么做”，不重复展开需求背景。
4. 测试文档要明确指回所验证的功能、需求、设计或计划。
5. 涉及总体架构、模块映射、持久化结构、公共 API、TOC 的变更，落地后必须同步更新 `docs/Toolbox-addon-design.md`。
6. `docs/FEATURES.md` 只记录稳定功能总览；功能尚未稳定时，不得用更新 `FEATURES.md` 代替功能/需求/设计/计划/测试文档。
7. 若某功能已有对应文档，新增需求澄清、重构方案、实施步骤或测试结果时，优先续写该功能的现有文档；只有主题边界发生实质变化时，才允许创建新的 `topic`。

---

## 8. 历史文档迁移规则

1. 旧目录中的文档若继续维护，优先迁移到新目录并按新命名补后缀。
2. 旧文档若暂不迁移，至少在下次续写时补齐元信息、状态与关联文档。
3. 已存在的日期型文件名属于历史写法，不要求一次性重命名；但新建文档不得继续沿用日期前缀。
4. 不要求一次性迁移所有历史文档；但从本规范生效后，不得继续新增旧式落点。

---

## 9. 禁止项

以下写法一律禁止：

1. 在功能文档中混入开发接入说明、代码示例或未来方案推导。
2. 在需求文档中混入完整设计方案或完整测试记录。
3. 在设计文档中用“待确认项”代替已经定下来的方案结论。
4. 在计划文档中只写笼统动作，如“实现功能”“补测试”，不写文件与验证方式。
5. 在测试文档中只写“已测试通过”，不写范围、步骤和结果。
6. 用 `docs/superpowers/specs/*.md`、`docs/superpowers/plans/*.md` 继续承接新需求或新计划。

新文档必须从以下模板复制：

- `docs/templates/spec-template.md`
