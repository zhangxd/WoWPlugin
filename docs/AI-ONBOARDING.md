# AI 协作开发：读档顺序与文档整理约定

面向「以后主要用 AI 加功能」时，减少重复说明、避免设计与实现漂移。

---

## 1. 每次加功能时，让 AI 必读（按顺序）

| 顺序 | 文件 | 作用 |
|------|------|------|
| ① | [Toolbox-addon-design.md](./Toolbox-addon-design.md) | **唯一总体方案**：架构、模块契约、数据约定、各能力边界；新功能必须能嵌进这套模型。 |
| ①′ | [AGENTS.md](../AGENTS.md) | **与 ① 同级必读**：领域对外 API、模块边界、**Lua 开发规范**、**暴雪 UI 挂接时机**；全文以该文件为准。 |
| ② | 本仓库根目录 `README.md`（若有） | 如何安装、TOC 名、支持的客户端；避免 AI 假设错游戏版本。 |
| ③ | **本次任务** | 用户消息里写清：模块名/行为/验收标准；若复杂，附 `docs/specs/` 下当期规格一节（见下）。 |

**可选、按需打开**

| 情况 | 再读 |
|------|------|
| 改核心加载/DB/注册表 | `Core/` 下实际文件 + 设计文档「2.1 通用架构」 |
| 只加一个模块 | `Modules/` 里**一个**现有模块当模板 + 设计文档「2.2 映射表」 |
| 与暴雪 Frame 名相关 | `docs/specs/` 或模块内注释里的白名单、版本备注 |
| **向暴雪 UI 挂子控件、改锚点、换 DataProvider** | [AGENTS.md](../AGENTS.md) **「暴雪 UI 挂接时机」**（`OnShow`、`hooksecurefunc` 等；勿以固定延迟为主路径）。 |
| **编写或修改 Lua** | [AGENTS.md](../AGENTS.md) **「Lua 开发规范」**。 |

**不要**让 AI 先通读整个 `Modules/` 所有文件，除非在做重构；**点名文件**更高效。

### 1.1 改动节奏（设计先行、审核、再编码）

触及模块行为、存档键、TOC、设置 UI 的修改，按三步：**对齐设计** → **审核文档**（总设计 + AGENTS）→ **需求方明确「开动」** 后改业务 Lua/TOC。规格可写在 `docs/specs/YYYY-MM-DD-<topic>.md`。  
未开动前不改业务代码；仅改文档与 `docs/**` 允许。细则见上文与 **[specs/2026-04-02-design-workflow-and-settings-groups.md](./specs/2026-04-02-design-workflow-and-settings-groups.md)**（协作节奏以本节与 AGENTS 为准；该文件 **§B** 为设置页分组需求）。

**例外**：错别字与**与行为无关**的注释；**hotfix**（提交说明注明原因）。与 [AGENTS.md](../AGENTS.md) 文首「改动节奏」一致。

### 1.2 AI 建议执行路径（需求从含糊到可落地）

本节约束 **AI（及按本仓库协作的人）** 在收到**未点名文件、未写验收标准**时的**推荐顺序**；与上文 **§1 必读顺序**、下文 **§3 最小信息包** 配套使用——**§3 是人应提供的输入**，**§1.2 是 AI 收到不完整输入时的默认行为**。

**何时视为「需求含糊」**

- 仅一句话（如「优化 tips」「修一下菜单」），无模块 id、无复现、无验收。
- 一词多义：`tips` 可能指 **GameTooltip 锚点/表现**、**设置页说明文案**、**某业务模块自绘提示**，或 **SavedVariables / 调试提示**。

**建议步骤（按顺序）**

1. **辨领域（先归类，再打开编辑器）**  
   - **浮动游戏提示**：`Toolbox.Tooltip`、`Modules/TooltipAnchor.lua`、`GameTooltip` / `hooksecurefunc`。  
   - **设置界面字符串**：`Toolbox/Core/Locales.lua` + 各模块 `RegisterSettings`；勿在业务里硬编码中文/英文句子。  
   - **聊天输出**：须经 **`Toolbox.Chat`**（见 AGENTS「门面优先」）。  
   - **暴雪窗口挂接/子控件**：AGENTS「暴雪 UI 挂接时机」；**禁止**以固定秒级 `C_Timer` 作为等布局的**唯一**手段。  
   - **模块专属玩法逻辑**：对应 `Modules/<name>.lua` + `ToolboxDB.modules.<moduleId>`；不先假设要改 `Core`。

2. **澄清再动手（若仍无法唯一映射到上述某一类）**  
   - 用**少量**（通常 1～3 个）封闭式问题让用户选方向，或请其补充 **§3** 中的验收与边界；**避免**在未对齐领域时大改多文件。

3. **读文档（执行 §1 表格）**  
   - 至少 **Toolbox-addon-design.md** + **AGENTS.md**；再按 §1「可选、按需打开」只读与本次任务相关的章节/文件。

4. **定位代码**  
   - 用搜索/语义检索锁定路径；**点名具体文件**阅读，**不要**默认通读整个 `Modules/`（除非重构或用户要求全景）。

5. **设计与开动（与 §1.1 一致）**  
   - 触及模块行为、存档、TOC、设置 UI：先对齐设计与文档，**开动**后再写业务代码；小范围 typo、与行为无关的注释除外。

6. **实现与收尾**  
   - 遵守 AGENTS：门面、Locales、注释、对外接口文档；改行为则按 **§4** 更新总设计等处。

**示例（映射练习）**

| 用户原话（含糊） | 应先辨别的方向 |
|------------------|----------------|
| 「优化 tips」 | 锚点/跟随/性能 vs 设置页文案 vs 某面板专用 `GameTooltip` |
| 「改设置」 | `SettingsHost` / `RegisterSettings` vs 仅 `Locales` 措辞 |
| 「菜单按钮没了」 | `Toolbox.GameMenu_Init` 与游戏菜单生命周期 vs 微型菜单模块 |

**与 [AGENTS.md](../AGENTS.md) 的关系**：根目录 AGENTS「先读文档」下列有**同一路径的摘要检查清单**（便于只开一份文件时速查）；**展开说明、示例与禁止项以本节为准**，两处若摘要与 §1.2 正文不一致，以 **§1.2** 为据并应修正 AGENTS 摘要。

## 2. 文档分层

```
docs/
├── Toolbox-addon-design.md   # 总设计：架构 + 约定 + 已有模块说明（随功能迭代更新「映射表」）
├── AI-ONBOARDING.md          # 本文件：读档顺序与文档分层
└── specs/                    # 可选：按日期或按功能的短期规格
    └── YYYY-MM-DD-<topic>.md # 一次大需求或复杂交互时写，实现后可合并回总设计或归档
```

- **总设计**：唯一长期架构；模块列表、扩展点、`ToolboxDB`、里程碑只维护这一份。
- **specs/**：当期需求与验收；实现后合并进总设计或删除过期内容。
- **Lua 注释与 Locales**：以 [AGENTS.md](../AGENTS.md) **「Lua 开发规范」** 与 **「项目约束」** 为准。

## 3. 加新功能时，人要补什么（给 AI 的最小信息包）

在对话里或 `docs/specs/xxx.md` 里尽量包含：

1. **模块 id**（如 `cooldown_overlay`）是否与 [设计文档映射表](./Toolbox-addon-design.md) 已预留一致。  
2. **客户端**：正式服（本仓库默认）。  
3. **数据**：存在 `ToolboxDB.modules.<id>` 哪些键；是否要迁移。  
4. **设置**：是否需要 `RegisterSettings`、大致控件（开关/滑条/下拉）。  
5. **边界**：不做什么（避免 AI 扩大范围）。  
6. **验收**：例如「重载后位置保留」「战斗中不报错」。

---

## 4. 功能做完后，文档谁更新

| 变更 | 更新哪里 |
|------|----------|
| 新模块上线 | `Toolbox-addon-design.md`：鸟瞰图、§2.2 映射表、§3 数据示例、§6 TOC、§7 里程碑 |
| 仅修 bug、未改行为 | 一般不改总设计；除非发现原约定错误，再改文档 |
| 架构级变更 | 先改 `Toolbox-addon-design.md`，再写代码 |

## 5. Cursor / 仓库根指针

实现约束以仓库根 **[AGENTS.md](../AGENTS.md)** 为准（与总设计、本文件并列）。

---

## 6. 不建议的做法

- 多份「总架构」互不同步（只保留 **Toolbox-addon-design.md** 为总纲）。  
- 把大段 API 教程复制进仓库（链到 [warcraft.wiki.gg](https://warcraft.wiki.gg) 或官方文档即可）。  
- 无日期、无主题的零碎 `notes.md` 长期堆在根目录。
