# docs 写作规范落地设计

- 文档类型：设计
- 状态：已确认
- 主题：docs-writing-standard
- 适用范围：`docs/**`
- 关联模块：无
- 关联文档：
  - `docs/AI-ONBOARDING.md`
  - `docs/FEATURES.md`
- 最后更新：2026-04-13

## 1. 背景

- 当前 `docs/**` 中同时存在 `features`、`specs`、`superpowers/specs`、`superpowers/plans` 等多种落点，同类文档的目录语义、命名和章节结构不稳定。
- 单次功能说明、设计决策、实施计划与测试记录存在混写现象，导致 AI 和人工在续写时难以判断“该写到哪里、该补什么、状态是什么”。

## 2. 设计目标

- 为 `docs/**` 建立单一权威写作规范。
- 统一功能、需求、设计、计划、测试五类文档的目录、命名、模板与状态。
- 将 `AI-ONBOARDING.md` 和入口文档回写为强制入口，避免只新增规范但没人遵循。

## 3. 非目标

- 不一次性迁移所有历史遗留文档。
- 不引入脚本化 lint、CI 校验或自动迁移工具。
- 不修改 `docs/**` 之外的协作文件。

## 4. 方案对比

### 4.1 方案 A：只新增规范文档

- 做法：新增一份规范文档，在 onboarding 中加一句“以后照此执行”。
- 优点：改动最少。
- 风险 / 缺点：目录职责不清、模板缺失，执行时仍会继续混写。

### 4.2 方案 B：规范 + 模板 + 目录边界

- 做法：新增权威规范文档、五类模板，并在 onboarding/入口文档中明确目录边界和强制规则。
- 优点：能够同时解决“写法不统一”和“放错目录”两类问题。
- 风险 / 缺点：需要先定义历史目录的处置方式。

### 4.3 方案 C：方案 B + 自动校验

- 做法：在方案 B 基础上再引入脚本或 CI，对文件名、目录、模板字段进行检查。
- 优点：门禁最强。
- 风险 / 缺点：当前问题先在文档层即可收口，引入自动化会扩大本次范围。

### 4.4 选型结论

- 选定方案：方案 B，并吸收状态流转与回写规则。
- 选择原因：先把口径、模板和目录收紧到可执行程度，避免规则落地成本过高。

## 5. 选定方案

- 新增 `docs/DOCS-STANDARD.md` 作为 `docs/**` 下功能、需求、设计、计划、测试文档的唯一写作规范。
- 新增 `docs/templates/` 下五类模板，统一新文档起稿格式。
- 明确新目录边界：`docs/features/`、`docs/specs/`、`docs/designs/`、`docs/plans/`、`docs/tests/`。
- 将 `docs/superpowers/specs/`、`docs/superpowers/plans/` 标记为历史遗留目录，不再新增同类文档。
- 明确同一功能在同一文档类型下只保留一份当前文档；重构、补充和状态推进均直接修改原文件，不另起平行文档。
- 回写 `docs/AI-ONBOARDING.md` 与 `docs/FEATURES.md`，让入口文档直接声明新规范。

## 6. 影响面

- 数据与存档：无。
- API 与模块边界：无。
- 文件与目录：新增 `DOCS-STANDARD.md`、`docs/templates/` 及规范目录；更新 onboarding 和功能总览入口。
- 文档回写：后续所有 `docs/**` 下需求、设计、计划、测试文档都按新规范执行。

## 7. 风险与回退

- 风险：历史文档仍会同时存在，短期内目录结构是“新旧并存”。
- 回退或缓解方式：通过规范明确“历史目录禁止新增，续写时优先迁移”，逐步收口而不做一次性大迁移。

## 8. 验证策略

- 检查仓库中已存在权威规范文档和五类模板。
- 检查 `AI-ONBOARDING.md` 已把该规范纳入读档路径和目录分层。
- 检查 `FEATURES.md` 已声明自身不是需求/设计/计划/测试文档。
- 检查同一功能在同一文档类型下是否继续沿用单文档续写，而不是新建平行文件。
