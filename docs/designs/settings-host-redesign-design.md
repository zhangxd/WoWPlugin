# 设置宿主重构设计

- 文档类型：设计
- 状态：已确认
- 主题：settings-host-redesign
- 适用范围：`Toolbox/UI/SettingsHost.lua` 的页内设置构建模型、现有模块设置控件映射、说明文字精简，以及基于当前 6 个叶子页的第二阶段交互重构
- 关联模块：`mover`、`tooltip_anchor`、`encounter_journal`、`quest`、`navigation`、`minimap_button`、`chat_notify`
- 关联文档：
  - `docs/specs/settings-host-redesign-spec.md`
  - `docs/plans/settings-host-redesign-plan.md`
  - `docs/Toolbox-addon-design.md`
- 最后更新：2026-05-01

> 本文承接 2026-04-29 已完成的纯叶子页结构重构，记录 2026-05-01 用户确认的第二阶段方案：在不改变 6 个叶子页、模块归属和存档语义的前提下，把页内布局与设置控件统一到更接近 Retail 新版 Settings 的交互口径。

## 1. 背景

- 第一阶段已经完成左侧树和叶子页职责收口，当前设置入口、默认打开规则和 Quest Inspector 承载位置已经稳定。
- 但页内控件仍大量沿用旧式模板与手工布局：
  - `InterfaceOptionsCheckButtonTemplate`
  - `UICheckButtonTemplate`
  - `UIDropDownMenuTemplate`
  - 多个勾选框手工互斥模拟单选
- 当前页面文字也偏多，常见形态是“模块简介 + 控件 + 独立 hint + 页尾公共区”，视觉密度不稳定，和现行 Retail 设置页的一致性不足。
- 用户本轮明确要求：保留现有叶子页和存档兼容性，但把页内交互与布局整体统一，并删除冗余文字说明。

## 2. 设计目标

- 保留 6 个叶子页和模块归属不变，只重构页内控件承载方式。
- 让普通布尔开关、单选、多选、下拉和操作按钮都进入同一套宿主级设置行模型。
- 用接近 Retail 新版设置页的“原生列表式”节奏替代旧的散排控件与长说明。
- 保持 `ToolboxDB` 键名、取值语义和模块行为不变。
- 保持现有刷新链路、即时生效时机和战斗内独立宿主模式不退化。

## 3. 非目标

- 不改左侧树分组，不新增页面层级，不回退到总览页模型。
- 不新增模块、不新增玩家可见入口、不修改 `Toolbox/Toolbox.toc`。
- 不重命名或重定义现有 `ToolboxDB` 键。
- 不借本轮重构新增控制台、仪表盘、主题皮肤或其它设置页外能力。
- 不顺带修改模块运行时逻辑，只改设置 UI 与设置绑定方式。

## 4. 方案对比

### 4.1 方案 A：原生列表式

- 做法：
  - 页面保持当前叶子页结构。
  - 页内统一改为“左侧标题与短说明，右侧控件”的列表式设置行。
  - 宿主提供统一 helper，模块通过 `RegisterSettings(box)` 声明设置项，不再直接拼旧控件。
- 优点：
  - 最接近 Retail 新版设置页的阅读和操作节奏。
  - 最容易统一普通开关、单选、多选和下拉的布局与间距。
  - 对现有 6 个叶子页和模块边界侵入最小。
- 风险 / 缺点：
  - 需要重做页内高度和滚动区计算。
  - 部分控件的展示方式会和旧版有视觉差异，例如 `chat_notify` 的颜色选择。

### 4.2 方案 B：分组卡片式

- 做法：
  - 继续保留当前叶子页结构。
  - 每个模块或功能区用卡片包裹，在卡片内部放设置项。
- 优点：
  - 模块边界清晰。
  - 视觉上更容易做“块状信息分组”。
- 风险 / 缺点：
  - 更像插件自定义控制台，不够贴近暴雪原生 Settings。
  - 页面内层级感更重，不符合用户想要的“最新设置操作方式”。

### 4.3 方案 C：摘要 + 详情式

- 做法：
  - 页面先展示设置摘要，再给详情区承载选中项。
  - 更接近配置器而不是传统设置页。
- 优点：
  - 复杂设置可以得到更强的详情承载能力。
- 风险 / 缺点：
  - 需要更大范围重构宿主和页面刷新逻辑。
  - 已超出“设置页交互统一”的必要范围。
  - 与 Retail 常规设置页的操作心智差异较大。

### 4.4 选型结论

- 选定方案：方案 A。
- 选择原因：
  - 与用户明确选中的视觉方向一致。
  - 在不改变叶子页结构和存档语义的前提下，能最大程度统一控件与布局。
  - 风险集中在宿主内部，便于控制影响面和回退范围。

## 5. 选定方案

### 5.1 页面节奏

所有叶子页沿用统一的原生列表式节奏：

1. 页面标题
2. 一句简短说明
3. 必要的分节标题
4. 设置行列表
5. 页底低频操作区

约束如下：

- 每个设置行只保留必要标题和一行内可读的短说明。
- 冗余长说明、重复说明和“控件下方再补一大段提示”的旧写法尽量移除。
- `debug`、`重置/重建`、打开工具面板等低频项放在页面靠后位置，不抢占首屏。

### 5.2 `box` 契约

继续保留 `RegisterSettings(box)` 作为模块协作面，但将 `box` 从普通 `Frame` 升级为宿主级设置构建容器。

宿主负责：

- 统一管理纵向游标和 `realHeight`
- 统一创建分节标题、设置行、说明文本和操作按钮
- 统一控件宽度、行高、左右间距和禁用态表现
- 统一承载语言、reload、页底公共低频动作区，以及模块通用开关区

模块负责：

- 通过宿主 helper 声明自身设置项
- 处理自身数据读写与行为回调
- 普通模块的 RegisterSettings(box) 只负责模块专属设置项，公共 enabled / debug / reset 区由宿主统一承载
- 仅在 helper 覆盖不了的区域保留自定义内容块，并显式回报高度

模块允许调用的 helper 类型：

- 分节标题
- 布尔开关行
- 单值选择行
- 多选列表
- 菜单 / 下拉选择行
- 操作按钮行
- 简短说明块

约束：

- 模块不再直接管理全页纵向游标。
- 宿主统一收口 `realHeight`。
- 自定义内容块仍可存在，但必须挂在宿主提供的布局锚点下，并明确向宿主回报实际高度。

### 5.3 控件映射规则

本轮统一映射如下：

- 普通布尔开关：
  - 改为统一 toggle 行
  - 左侧标题 + 短说明，右侧开关
- 两项单选：
  - 优先改为 segmented / pill 样式
  - 不再使用两个勾选框手工互斥
- 三项及以上单选：
  - 优先改为 dropdown / menu button 样式
  - 不再使用多项勾选框手工互斥
- 多选项：
  - 改为统一多选列表
  - 行间距、勾选态和禁用态由宿主统一
- 下拉选择：
  - 不再继续使用 `UIDropDownMenu_*`
  - 走宿主菜单按钮 + 弹层列表的等价封装，保持与现行 Settings 菜单选择一致的操作语义
- 操作按钮：
  - 以设置行或页底动作区形式呈现，不再和其它控件散排

### 5.4 当前模块映射

当前最小迁移面与落点如下：

- `SettingsHost` 语言选择：
  - 从互斥勾选改为单值选择控件
- `chat_notify`：
  - 两个颜色选择从 `UIDropDownMenuTemplate` 改为统一菜单按钮
- `tooltip_anchor`：
  - `default / cursor / follow` 从勾选框互斥改为单值选择控件
- `mover`：
  - `titlebar / titlebar_empty` 从勾选框互斥改为双项 segmented 选择
  - `allowDragInCombat` 纳入统一 toggle 行
- `minimap_button`：
  - `showMinimapButton`、`showCoordsOnMinimap` 纳入统一 toggle 行
  - 坐标锚点从双勾选互斥改为双项 segmented 选择
  - flyout 项改为统一多选列表
- `quest`、`encounter_journal`、`navigation`：
  - 没有复杂选项时，仍统一到新的 toggle / action / note 节奏

### 5.5 刷新与依赖策略

- 行级依赖由宿主 helper 显式表达，例如“某父开关关闭时，子项整行禁用”。
- 页面注册只在宿主初始化阶段执行一次，后续重建不重复注册类目或叶子页。
- 语言切换统一走 `RefreshAllPages()` 全量重建。
- 仅影响勾选态、文案或禁用态时，优先局部刷新当前行或当前组。
- 影响区块显隐、行数、`realHeight` 或自定义内容块高度时，统一走 `BuildPage()` 重建当前叶子页。
- 现有模块的 `OnEnabledSettingChanged`、`OnDebugSettingChanged`、页面重建和即时生效时机保持不变。
- `MinimapButton` 和 `Quest` 当前较强的页面重建依赖继续保留，只改触发入口，不改行为语义。

### 5.6 异常值回退策略

- 单选 / 下拉读取到未知值时，UI 不保留占位态，而是直接回退到该设置的默认项。
- 页面构建阶段即完成归一化，并将非法值回写为默认值，避免出现双选中、无选中或“看起来选中 A、实际存档是 B”的状态。
- `chat_notify` 若新菜单按钮不适合继续展示彩色文字，可退一步改为“颜色名称 + 当前效果预览”，但底层值语义不变。

### 5.7 战斗内独立宿主模式

- 战斗内独立宿主模式仍是本轮必保路径。
- 打开优先级为：显式目标叶子页 > 最近一次有效的 `settingsLastLeafPage` > `通用`。
- 新的设置行、菜单按钮和单选交互在该模式下必须保持等价可操作。
- 战斗内切换叶子页时，允许在已创建的独立宿主内部切页，并在切页成功后更新 `settingsLastLeafPage`。
- 该路径不得重新注册类目，也不得调用系统 `Settings.OpenToCategory`。
- 若某原生控件路径只在系统 Settings 页面中稳定，宿主需要在独立模式提供等价封装，而不是退回到只读或不可操作状态。
- 验证阶段需要单独覆盖该模式，而不是默认系统设置页通过即可视为完成。
### 5.8 文案策略

- 页面说明压缩为“一句话说明 + 必要分节标题”。
- 设置行只保留必要的短说明。
- 删除对阅读路径帮助不大的重复解释。
- 模块行为语义不清的地方，优先把说明贴近设置行，而不是单独堆成长段文案。

## 6. 影响面

- 数据与存档：
  - `ToolboxDB.modules.*` 与 `ToolboxDB.global` 的现有键名和取值语义保持不变。
- API 与模块边界：
  - 模块运行时行为不变。
  - `SettingsHost` 进一步从“叶子页组合宿主”提升为“统一设置控件宿主”。
  - `RegisterSettings(box)` 契约继续保留，但 `box` 能力升级且职责更明确。
- 文件与目录：
  - 核心改动集中在 `Toolbox/UI/SettingsHost.lua`
  - 预计需要跟随调整：
    - `Toolbox/Modules/ChatNotify.lua`
    - `Toolbox/Modules/TooltipAnchor.lua`
    - `Toolbox/Modules/Mover.lua`
    - `Toolbox/Modules/MinimapButton.lua`
    - `Toolbox/Modules/Quest.lua`
    - `Toolbox/Modules/EncounterJournal.lua`
    - `Toolbox/Modules/Navigation.lua`
    - `Toolbox/Core/Foundation/Locales.lua`
    - `tests/validate_settings_subcategories.py`
- 文档回写：
  - 落地后需更新 `docs/Toolbox-addon-design.md`
  - 如功能说明里的设置项描述发生变化，也要同步对应 `docs/features/*.md`

## 7. 风险与回退

- 风险：
  - `yOffset` 与 `realHeight` 重构后容易出现滚动高度不准、页尾被截断或控件重叠。
  - `MinimapButton` 和 `Quest` 的重建链路如果适配不当，可能出现设置后不刷新或过度刷新。
  - `chat_notify` 的颜色选择迁移后，若视觉反馈不足，可能降低可读性。
  - 战斗内独立宿主模式若直接照搬系统设置控件，可能出现可见但不可稳定交互的退化。
- 回退或缓解方式：
  - 先在宿主中建立统一 helper，再逐个模块迁移控件，避免一次性改完整个页面。
  - 迁移过程中保留原有行为回调，只替换控件承载方式。
  - 若某一类控件迁移效果不理想，可局部回退到上一阶段的页面结构，再单独调整该控件类型。
  - 若独立宿主模式验证不过，优先为该模式补等价封装，而不是整体放弃控件统一。

## 8. 验证策略

实现落地后至少验证以下结果：

1. 所有 6 个叶子页仍可正常打开，默认打开规则不变。
2. 页内控件整体呈现为原生列表式节奏。
3. 原有互斥勾选模拟单选与 `UIDropDownMenu_*` 旧链路已移除。
4. `RegisterSettings(box)` 的宿主 / 模块职责边界在实现中得到体现。
5. 所有现有设置都可修改、保存，并在 `/reload` 后保留原值。
6. 单选项不会出现双选中或无选中的非法状态；非法枚举值会回退并归一到默认项。
7. 相关模块保持原有即时生效或原有生效时机。
8. 页面滚动区正常，无明显截断、重叠或间距错乱。
9. 战斗内独立宿主模式下，新控件仍可等价操作。
10. `python tests/run_all.py --ci` 通过。

建议验证方式：

- 静态校验：补或改设置页结构与旧控件使用口径校验
- 自动化验证：`python tests/run_all.py --ci`
- 游戏内验证：检查页面滚动、入口一致性、即时生效、语言切换和战斗内独立宿主模式
